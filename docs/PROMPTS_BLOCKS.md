# Компонентные промты (blocks v2)

Единая схема мастер-промтов для всех нод пайплайна: каждый шаг собирается
из шаблона (`steps/<step_id>/template.md`) + переиспользуемых блоков
(`blocks/<category>/<name>.md`) + переменных (`{{VAR:NAME}}`). Всё хранится
**файлами на диске** в `prompts/` — ничего не «зашито» в код, редактировать
можно и вручную (git), и через Studio UI (REST + вкладка «Промты GPT»).

## 1. Структура на диске

```
prompts/
  _vars.md                    # справочник {{VAR:...}} и их дефолтов
  01_plan/ … 07_animation/     # legacy: мастер-промты «одним файлом» (default.md + варианты)
  blocks/<category>/<name>.md  # переиспользуемые фрагменты («мир», «стиль», «свет», ...)
  steps/<step_id>/template.md  # шаблон шага — собирает блоки + vars в 5–7 смысловых блоков
  styles/<preset>.json         # пресеты: набор блоков + vars одним кликом
```

Оба слоя («legacy» одним файлом и «steps/blocks v2») сосуществуют:
какой используется для конкретного проекта, решает
`Project.prompt_overrides["use_blocks_v2"]` (или непустой
`prompt_overrides["blocks"]`) — см. `prompt_composer.project_uses_blocks_v2()`.

## 2. Формат шаблона шага: 5–7 блоков

Каждый `steps/<step_id>/template.md` — это markdown с заголовками `## N. …`,
их должно быть от 5 до 7. Порядок фиксирован по смыслу:

1. **`## 1. ТЕХНИЧЕСКАЯ ЧАСТЬ`** — обязателен всегда, первым. Пишется
   статическим текстом (не через `{{BLOCK:}}`, т.к. специфичен для шага) и
   строго отвечает на три вопроса:
   - **откуда читаю** — какие файлы/ячейки/входные данные приходят в этот шаг;
   - **куда пишу** — какой файл/лист/строка/колонка получает результат,
     и в каком формате (xlsx / txt / картинка);
   - **на что обратить внимание** — валидация, лимиты, что нельзя ломать
     (переименовывать листы, обрезать файл и т.п.).
2. **РОЛЬ И ЗАДАЧА** — кто модель и что конкретно надо сделать.
3–6. **Контентные блоки** — тема/мир, стиль/тон, структура/композиция,
   запреты — набор зависит от шага (не все шаги используют все категории).
7. **ФОРМАТ ВЫВОДА** — объём, лимиты длины, что именно приложить к ответу.

Технический блок 1 намеренно не «блок» в смысле `{{BLOCK:cat}}` (его нельзя
менять пресетом или весом) — потому что это протокол ввода-вывода шага, а
не творческое содержимое. Все технические детали в нём взяты из реальной
логики раннеров (`app/services/xlsx_step_runners.py`,
`app/services/chatgpt_xlsx.py`), не выдуманы.

Проверяется автоматически: `tests/test_prompt_composer.py` гоняет по всем
`steps/*/template.md` и требует 5–7 заголовков `## N.` и наличие блока
«ТЕХНИЧЕСКАЯ ЧАСТЬ» первым.

## 3. Сборка: `{{BLOCK:cat}}` и `{{VAR:NAME}}`

`app/services/prompt_composer.py`:

- `{{BLOCK:category}}` → контент из `blocks/<category>/<name>.md`
  (какое `<name>` — решает `prompt_overrides.blocks[category]`, дефолт
  в `DEFAULT_BLOCKS`).
- `{{VAR:NAME}}` → подстановка из merged vars (project overrides + style
  preset + `DEFAULT_VARS`); подставляется **после** сборки блоков, поэтому
  `{{VAR:...}}` работает и внутри своего текста блока (см. §4).

## 4. Значение блока: имя файла, вес 0–1 или свой текст

`prompt_overrides.blocks[category]` поддерживает **три равнозначные
формы** (полная обратная совместимость — старые проекты с чистыми строками
продолжают работать без изменений):

```json
{
  "world": "cats_anthropomorphic",                                 // как раньше: имя файла, вес 1.0
  "visual_style": {"name": "micro_pixelart", "weight": 0.6},        // имя файла + вес
  "lighting": {"text": "мягкий свет слева, {{VAR:ASPECT_RATIO_VIDEO}}", "weight": 1} // свой текст вместо файла
}
```

- **`weight`** (0.0–1.0, дефолт 1.0) — это не числовой параметр для модели
  (текстовые LLM не умеют веса как diffusion-модели), а понятная текстовая
  метка приоритета, которую `prompt_composer._weight_prefix()` добавляет
  перед контентом блока, если вес отличается от 1.0:
  - `≥ 0.7` → «[ВАЖНО, соблюдай строго]»
  - `0.4–0.7` → «[учитывай наравне с другими блоками]»
  - `< 0.4` → «[фоновый акцент, второстепенно]»
  - `= 1.0` (дефолт) → без пометки, чистый текст блока как раньше.
- **`text`** — свободный текст вместо выбора файла из библиотеки. В нём
  можно использовать те же «ячейки» `{{VAR:ИМЯ}}`, что и в обычных
  шаблонах/блоках — они подставляются на общем проходе VAR-подстановки,
  ничего специального делать не нужно.
- Разбор значения (`resolve_block_value()`) никогда не бросает исключение:
  любая некорректная форма (битый вес, отсутствующий файл, `None`) тихо
  превращается в дефолт или в диагностический html-комментарий
  (`<!-- missing block: cat -->`) прямо в тексте промта — так баг видно
  сразу в превью, а не как падение сборки.

## 5. UI Studio

Студия ноды (правая панель) → вкладка **«Промты GPT»**:

- **«Файлы промтов»** (`prompt-files-panel.tsx`) — выбор/редактирование
  `.md`-варианта мастер-промта на диске (legacy-слой).
- **«Блоки промта — вес и содержимое»** (`blocks-weight-panel.tsx`, новое) —
  показывается только если у ноды есть `steps/<id>/template.md` **и** этот
  шаблон реально использует хотя бы одну категорию `{{BLOCK:}}`
  (`GET /api/prompt-studio/catalog` → `step_block_categories[step_id]`).
  Для каждой такой категории — выбор файла из библиотеки **или** свой
  текст, плюс слайдер веса 0–1. Сохраняется сразу в
  `project.prompt_overrides.blocks` — то самое поле, которое реально
  читает `compose_step()` (без промежуточных слоёв, которые не доходят до
  сборки промта).
- Категории, которых нет в шаблоне шага, никогда не показываются — так
  невозможно настроить параметр, который ни на что не влияет.

## 6. API

- `GET /api/prompt-studio/catalog` — `block_categories`,
  `step_block_categories` (категории по каждому шагу), `steps`,
  `node_type_to_step`, `style_presets`.
- `POST /api/prompt-studio/compose` — превью сборки (без сохранения).
- `PATCH /api/prompt-studio/projects/{id}/prompt-config` — сохранить
  `blocks` / `vars` / `style_profile` в `prompt_overrides` проекта.
- `GET/PUT/DELETE /api/prompt-studio/projects/{id}/gpt-text/{step_code}` —
  сопроводительный текст в чат (отдельно от мастер-промта, см. `chatgpt_xlsx.py`).

## 7. Файловое хранение (важно для «невозможности ошибок»)

Всё, что участвует в сборке промта, — обычные файлы на диске проекта:

| Что | Где |
|---|---|
| Мастер-промты (все варианты) | `prompts/<step>/*.md` (legacy) и `prompts/steps/<id>/template.md` (v2) |
| Блоки | `prompts/blocks/<category>/<name>.md` |
| Пресеты стилей | `prompts/styles/*.json` |
| Выбор юзера для проекта | SQLite `Project.prompt_overrides` (JSON), зеркалится в `project.xlsx` как читаемые значения полей — не в отдельный файл |
| Временные файлы, уходящие в ChatGPT | `data/videos/<slug>/tmp_gpt/prompt_*.txt` |
| Результат каждого шага | `data/videos/<slug>/project.xlsx` (+ `voiceover.txt`, `characters/`, `items/`) |
| Снапшот промтов для mass-factory | `data/batches/<slug>/prompts/` (копия всей `prompts/` на момент создания батча) |

Ни один из шагов не хранит логи/промты только в памяти — при перезапуске
воркера ничего не теряется (кроме WebSocket-стрима логов в
`app/services/event_bus.py`, который намеренно эфемерный — для истории
статусов используется `Attempt.logs` в SQLite).

## 8. Известные ограничения / follow-up

- `04_hero_style` (пикер стиля для Hero) пока не участвует в
  `NODE_TYPE_TO_STEP` — использует только legacy-слой файлов, blocks v2
  для него не собирается.
- `meta.prompt_styles` / `custom_styles` (`prompt-style-panel.tsx`) — более
  ранний, независимый прототип персональных пресетов на уровне
  ноды-в-графе (а не всего проекта); он не читается `compose_step()` и
  существует отдельно от `prompt_overrides.blocks`. Использовать для
  реальной настройки промта следует именно панель из §5.

## 9. Без Telegram

В `.env`:

```env
TELEGRAM_ENABLED=false
TELEGRAM_BOT_TOKEN=
```

Запуск: `.\start-studio.ps1` или `python -m app.main` — воркер + API, HITL через веб.
Шаги: `POST /api/projects/{id}/steps/plan/run` или кнопка «Создать Run» на графе.
