ПРИМЕЧАНИЕ: Этот агент — не GPT, создающий текст. Это операционный runbook
для автоматического пайплайна (Python/bash скрипт) или для оператора,
который запускает генерацию видеоклипов в Outsee и обновляет статусы в Excel.
Если ты — GPT, читающий этот файл для контекста: §3 описывает входные данные,
§8 — статус R66 который должен появиться после работы этого шага.

§1. РОЛЬ И НАЗНАЧЕНИЕ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Этот шаг отвечает за:
  1. Считывание промтов анимации (R48/R64/R65) из листа «план»
  2. Считывание референсных картинок (R45/R46/R47) как стартового кадра
  3. Отправку каждого промта+картинки в Outsee (image-to-video)
  4. Скачивание готовых клипов и сохранение в папку серии
  5. Обновление статуса R66: «видео_ок» или «переделать»
  6. Сигнал для человека о готовности клипов к просмотру

§2. МЕСТО В ПАЙПЛАЙНЕ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ПРЕДЫДУЩИЕ ШАГИ:
  ← ser_12_image_prompts : R45/R46/R47 (промты картинок) заполнены
  ← ser_images           : картинки сгенерированы, R66 = «картинки_ок» для одобренных
  ← ser_13_anim_prompts  : R48/R64/R65 (промты видео) заполнены
  ← человек              : одобрил галерею картинок (стоп-кран)

ЭТОТ ШАГ:
  → Читает R48/R64/R65 и референсные картинки.
  → Генерирует клипы ≤8 сек.
  → Сохраняет файлы.
  → Обновляет R66: «видео_ок».

СЛЕДУЮЩИЕ ШАГИ (параллельно после одобрения):
  → ser_14_dialogue_audio : разметка речи
  → ser_15_music_sfx      : разметка музыки/шумов
  → ser_16_assemble       : черновой монтаж

ЧЕЛОВЕЧЕСКИЙ СТОП-КРАН:
  После R66 = «видео_ок» → уведомление человеку.
  Человек просматривает клипы по сценам.
  Принятые: R66 = «видео_ок» остаётся.
  Отклонённые: R66 = «переделать» (возможно с новым промтом от ser_13).
  После одобрения → запуск параллельного ser_14 + ser_15 + ser_16.

§3. ВХОДНЫЕ ДАННЫЕ (ЧТО ЧИТАЕМ)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Файл: project.xlsx (серия)
  Лист: «план»
  Строки видео-промтов: R48 (кадр1), R64 (кадр2), R65 (кадр3)
  Референсные картинки (соответственно): R45, R46, R47
  Пути к картинкам: data/series/{series_id}/images/{scene_id}_{shot_id}_{R4X}.jpg

  Фильтрация: только ячейки, где:
    - R48/R64/R65 непустые
    - R66 для данного кадра = «картинки_ок» (необходима одобренная картинка)
    - R66 для данного кадра ≠ «видео_ок» (не перегенерировать одобренные)
    - R67 ≠ «да» (не трогать кадры после picture lock)

  Метаданные:
    - id scene, id shot (для именования файлов)
    - R50 «время на кадр» — целевая длительность (≤8 сек)

§4. ПОРЯДОК ГЕНЕРАЦИИ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ШАГ 1. Открыть Excel, прочитать «план», собрать очередь:
  Список (видео-промт, путь_к_картинке, id сцены, id shot, R4X, R50)

ШАГ 2. Проверить наличие картинки:
  Если файл картинки не существует:
    → Пропустить этот клип, записать в лог «картинка не найдена».
    → Если R66 = «картинки_ок» но файл пропал — пометить «переделать» (картинку).

ШАГ 3. Для каждого промта в очереди:
  3.1. Отправить в Outsee: image-to-video запрос.
       Входные данные: стартовый кадр (JPG) + промт движения (R48/R64/R65).
       Длительность: из R50 или 6 сек по умолчанию (≤8 сек).
  3.2. Дождаться результата (таймаут: 180 сек — видео дольше картинки).
  3.3. Скачать MP4/MOV.
  3.4. Сохранить в: data/series/{series_id}/clips/{scene_id}_{shot_id}_{R_row}.mp4
       Пример: data/series/e01/clips/scene_03_shot_01_R48.mp4
  3.5. Обновить R66:
       - Успех: «видео_ок»
       - Ошибка: «переделать»

ШАГ 4. После обработки всей очереди:
  4.1. Сохранить Excel.
  4.2. Уведомление: «Клипы готовы: N успешно, M ошибок».

§5. МЯГКИЙ ПОВТОР (SOFT RETRY) ДЛЯ ВИДЕО
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Видеогенерация медленнее картинок — таймауты дольше, CDP-сессии нестабильнее.

При ошибке одного клипа:
  ПОПЫТКА 1: Ждать 10 сек, повторить.
  ПОПЫТКА 2: Ждать 30 сек, повторить.
  ПОПЫТКА 3: Ждать 60 сек, повторить.
  ПОПЫТКА 4: Пометить «переделать», перейти к следующему.

КРИТИЧЕСКИ ВАЖНО: При CDP timeout на видео
  → НЕ удалять уже сгенерированные клипы других кадров серии.
  → НЕ сбрасывать R66 = «видео_ок» для ранее успешных клипов.
  → Не откатывать R66 = «картинки_ок» — картинки остаются как есть.
  → Пересоединиться с CDP, продолжить очередь с места сбоя.

При прерванном запуске:
  → Пропускать клипы с R66 = «видео_ок».
  → Генерировать только «переделать» и пропущенные.

§7. ПАРАЛЛЕЛЬНОСТЬ И КВОТА
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Видеогенерация значительно тяжелее картинок:
  - По умолчанию: 1 запрос одновременно (sequential).
  - Задержка между запросами: 5–10 сек.
  - Каждый клип: 20–120 сек обработки в Outsee.

Для серии из 50 ракурсов × 3 кадра = до 150 клипов:
  Реалистичное время: 2–4 часа при sequential.
  Рекомендация: запускать на ночь или на период ожидания.
  Прогресс: уведомлять каждые 5 клипов.

§8. СТАТУС R66 ДЛЯ ВИДЕО — ДЕТАЛЬНОЕ ОПИСАНИЕ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

R66 общая строка листа «план» для столбца.
Видео-статусы накапливаются на той же строке (или на отдельной строке R66b).
Конвенция: если картинка и видео оба «ок» — R66 = «видео_ок».
Если картинка ок, видео нет: R66 = «картинки_ок» (видео не готово).
Если оба провалились: R66 = «переделать».

Значения для видео:
  «картинки_ок»        : картинка одобрена, видео ещё не генерировалось
  «видео_генерируется» : Outsee обрабатывает
  «видео_ок»           : клип получен и сохранён ← основной успешный статус
  «переделать»         : ошибка или отклонено человеком

ser_16_assemble читает R66 = «видео_ок» для заполнения поля файл_видео в Монтаже.

§9. ВЗАИМОДЕЙСТВИЕ С OUTSEE (CDP, image-to-video)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Outsee поддерживает image-to-video: загрузить картинку → промт → видео.

Шаги:
  1. Подключиться к Chrome via CDP (localhost:29229)
  2. Навигировать на страницу image-to-video в outsee.io
  3. Загрузить референсную картинку (JPG из R45/R46/R47 путей)
  4. Вставить промт движения (R48/R64/R65)
  5. Установить длительность (из R50 или 6 сек default, ≤8 сек max)
  6. Запустить генерацию
  7. Polling каждые 5 сек (видео долго обрабатывается)
  8. Скачать MP4

При ошибке CDP (timeout, disconnect):
  - Reconnect: до 3 попыток
  - После 3 провалов: пометить клип «переделать», продолжить очередь
  - Лог: записать детали ошибки

§11. ЧЕЛОВЕЧЕСКИЙ СТОП-КРАН: ПРОСМОТР КЛИПОВ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

После завершения генерации:
  1. Studio UI показывает галерею клипов по сценам (mp4 с плеером).
  2. Человек для каждого клипа:
     «принять» → R66 = «видео_ок» остаётся
     «переделать» → R66 = «переделать»
       (→ ser_13 пишет новый R48/R64/R65, пайплайн перегенерирует только этот клип)
  3. После одобрения всех включаемых клипов → нажать «Запустить монтаж».
  4. Пайплайн параллельно запускает ser_14 + ser_15 + ser_16.

Пайплайн НЕ запускает монтаж автоматически без явного сигнала.

§12. ДИАГНОСТИКА ЧАСТЫХ ПРОБЛЕМ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ПРОБЛЕМА: Клипы «дрожат» сильнее, чем нужно
  Причина: Outsee применяет handheld по умолчанию.
  Решение: В промте (R48) явно написать «static camera» или «no camera shake».
    ser_13 должен указывать тип движения. Переписать R48 через ser_13.

ПРОБЛЕМА: Клип не соответствует стартовому кадру (другие цвета, персонаж)
  Причина: Outsee не «взял» картинку как референс.
  Решение: Проверить путь к картинке. Убедиться, что файл JPG не повреждён.
    Попробовать пересохранить картинку в стандартном разрешении (1080×1920).

ПРОБЛЕМА: Клип длиннее 8 секунд
  Причина: Outsee проигнорировал параметр длительности.
  Решение: Принять и обрезать при монтаже. Пометить в заметках Монтажа.
    Для следующей генерации: указывать длительность явнее в промте («~6s»).

ПРОБЛЕМА: Slow push-in у всех клипов (дефолт Outsee)
  Причина: Outsee применяет slow push без явной команды иного.
  Решение: В промте R48 явно указать «static camera» для статичных кадров.
    ser_13 должен включать движение в каждый промт. Проверить ser_13 вывод.

ПРОБЛЕМА: CDP timeout каждые 5 клипов
  Причина: Нестабильное соединение или Chrome переполнен памятью.
  Решение: Перезапустить Chrome между batches по 10 клипов.
    Добавить gc.collect() и очистку CDP-сессии каждые N клипов.

ПРОБЛЕМА: «переделать» у всех кадров одной сцены
  Причина: Скорее всего R48/R64/R65 пустые для этой сцены (ser_13 не заполнил).
  Решение: Запустить ser_13 заново для этих столбцов, потом повторить генерацию.
